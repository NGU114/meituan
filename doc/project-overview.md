# 仿美团外卖项目目标与原型说明

## 1. 项目名称

仿美团外卖系统

## 2. 项目背景

本项目是一个面向课程设计或毕业设计的外卖业务系统，目标不是完整复刻真实美团外卖，而是实现一个多角色协同、业务流程完整、能够稳定演示的简化版外卖平台。

项目围绕“用户下单、商家接单、骑手配送、管理员管理”这一核心业务展开，重点体现前后端分离开发、订单流转设计、角色权限控制和业务闭环实现能力。

## 3. 项目目标

本项目的总体目标是搭建一个简化版外卖平台，使四类角色可以在同一套系统中完成一条完整的订单闭环流程。

核心目标如下：

- 用户能够注册、登录、浏览店铺、选择商品、加入购物车、提交订单、模拟支付、查看订单、确认收货并评价。
- 商家能够维护店铺信息、管理商品、接收订单、更新制作状态、将订单流转到待配送。
- 骑手能够查看待配送订单、接单、配送、完成订单。
- 管理员能够查看用户、店铺、订单和平台基础统计数据。

项目最终应支持如下完整链路：

```text
用户下单 -> 商家接单 -> 商家出餐 -> 骑手接单配送 -> 用户确认收货 -> 用户评价 -> 管理员查看订单与统计数据
```

## 4. 项目定位

本项目定位为“课程设计/毕设版 MVP 外卖系统”。

这里的 MVP 指最小可行产品，即第一版不追求真实商业平台的全部复杂度，而是优先完成业务主流程，确保系统能够真正跑通并演示。

因此，本项目强调：

- 先完成业务闭环，再考虑功能扩展
- 先保证接口可联调，再优化页面表现
- 先保证状态流转正确，再补充增强功能

## 5. 项目原型范围

第一版建议只实现以下 4 个端：

### 5.1 用户端

用户端主要面向普通点餐用户，负责完成下单全过程。

建议页面包括：

- 登录页
- 注册页
- 首页/店铺列表页
- 店铺详情页
- 购物车页
- 确认订单页
- 支付结果页
- 我的订单页
- 订单详情页
- 地址管理页
- 评价页

### 5.2 商家端

商家端负责店铺经营和订单处理。

建议页面包括：

- 登录页
- 店铺信息管理页
- 商品管理页
- 订单列表页
- 订单详情页
- 接单/制作中/待配送操作页

### 5.3 骑手端

骑手端负责查看配送任务并完成配送。

建议页面包括：

- 登录页
- 可接订单页
- 我的配送页
- 配送详情页
- 送达操作页

### 5.4 管理端

管理端负责平台基础管理与数据查看。

建议页面包括：

- 登录页
- 仪表盘
- 用户管理页
- 店铺管理页
- 订单管理页
- 统计分析页

## 6. 核心业务流程

整个系统的核心不是页面，而是订单状态流转。

建议第一版订单状态设计如下：

- `PENDING_PAYMENT`：待支付
- `PAID`：已支付
- `ACCEPTED`：已接单
- `PREPARING`：制作中
- `READY_FOR_DELIVERY`：待配送
- `DELIVERING`：配送中
- `COMPLETED`：已完成
- `CANCELLED`：已取消

对应业务流转为：

```text
待支付 -> 已支付 -> 已接单 -> 制作中 -> 待配送 -> 配送中 -> 已完成
```

取消或异常情况下流转为：

```text
待支付/已支付 -> 已取消
```

从系统设计角度看，整套项目本质上就是不同角色围绕订单状态不断进行操作和推进。

## 7. 各角色功能目标

### 7.1 用户角色

用户角色应实现以下功能：

- 注册与登录
- 查看店铺列表
- 查看店铺详情和菜品信息
- 加入购物车
- 管理收货地址
- 提交订单
- 模拟支付
- 查看订单列表
- 查看订单详情
- 确认收货
- 评价订单

### 7.2 商家角色

商家角色应实现以下功能：

- 登录
- 查看和修改店铺信息
- 查看商品列表
- 新增商品
- 修改商品
- 上下架商品
- 查看订单列表
- 接单
- 标记制作中
- 标记待配送

### 7.3 骑手角色

骑手角色应实现以下功能：

- 登录
- 查看可接订单
- 接单
- 查看我的配送订单
- 更新配送状态
- 送达订单

### 7.4 管理员角色

管理员角色应实现以下功能：

- 登录
- 查看平台仪表盘
- 查看用户列表
- 查看店铺列表
- 查看订单列表
- 查看订单详情
- 查看平台基础统计数据

## 8. 第一版不做的内容

为了保证项目可落地，第一版建议不做以下复杂功能：

- 第三方真实支付
- 地图定位与导航
- 智能派单
- 搜索推荐算法
- 优惠券、满减、红包等复杂营销功能
- 秒杀和高并发抢购
- 微服务拆分

这些内容可以作为扩展项或答辩时的后续优化方向，不建议在第一版中强行实现。

## 9. 技术目标

本项目除了业务目标外，还应体现以下技术目标：

- 具备清晰的前后端分离结构
- 具备多角色权限控制
- 具备规范的接口设计
- 具备可复现的订单状态流转
- 具备基本的数据持久化能力
- 具备基础异常处理与统一返回格式
- 具备可演示、可联调、可测试的完整后端服务

## 10. 项目验收目标

项目验收时，至少应完成以下完整演示流程：

1. 用户登录系统并选择店铺、菜品加入购物车
2. 用户提交订单并完成模拟支付
3. 商家端查看新订单并完成接单
4. 商家端将订单状态改为制作中、待配送
5. 骑手端查看待接订单并接单配送
6. 用户端查看配送状态并确认收货
7. 用户完成订单评价
8. 管理员后台能够查询该订单并看到统计数据变化

只要以上流程可以完整演示，并且数据状态一致，就说明项目的核心目标已经实现。

## 11. 项目最终成果形式

项目最终建议形成以下成果：

- 一套完整运行的后端服务
- 用户端页面
- 商家端页面
- 骑手端页面
- 管理端页面
- 数据库设计文档
- 接口文档
- 项目目标与原型说明文档
- 演示账号和演示脚本

## 12. 总结

本项目不是要完全复刻真实商业级美团外卖，而是要完成一个具备真实业务逻辑的“简化版外卖平台”。

其核心目标可以归纳为：

- 多角色协同
- 订单流程完整
- 前后端可联调
- 数据状态正确
- 系统能够稳定演示

因此，本项目最重要的不是页面有多复杂，而是是否真正打通了“下单到完成”的业务闭环。
